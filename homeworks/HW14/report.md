# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- База знаний: **20 справочных статей по концепциям машинного обучения** (ML glossary) — каждый документ объясняет один термин/метод.
- Чанкинг: символьный с перекрытием (`chunk_size=200`, `overlap=40`), итого **47 чанков** из 20 документов.
- Embedding-модель: `all-MiniLM-L6-v2` (sentence-transformers), размерность 384.
- Индекс: `FAISS IndexFlatIP` (косинусное сходство через нормализованные векторы).
- Retrieval на 12 контрольных запросах показал идеальный результат: `hit@3 = 1.000`, `recall@3 = 1.000`, `MRR@3 = 1.000`.
- После добавления 5 новых документов и переиндексации retrieval изменился во всех тестовых запросах.
- Mini-RAG собран: извлечение контекста + шаблонный ответ с указанием источников.

## 2. Среда и воспроизводимость

- Device: CPU
- faiss: 1.13.2
- sentence-transformers + torch (внутри модели)
- Seed: 42 (random, numpy, torch)
- Запуск: `HW14.ipynb` → Run All

## 3. База знаний и подготовка документов

- Предметная область: базовые концепции машинного обучения (ML glossary).
- Число документов: 20 (исходных) + 5 (после обновления) = 25.
- Число чанков: **47** (исходных) → **~60** после добавления 5 документов.
- Способ чанкинга: простой символьный с перекрытием.
- Параметры: `chunk_size=200`, `overlap=40`.
- Комментарий: База состоит из коротких, тематически изолированных статей. Каждый документ посвящён одной концепции, что делает retrieval задачу удобной и предсказуемой. Перекрытие в 40 символов помогает сохранять контекст на границах чанков.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс
- Модель эмбеддингов: `all-MiniLM-L6-v2`
- Метрика: косинусное сходство (`IndexFlatIP` + нормализованные векторы)
- Индекс: `faiss.IndexFlatIP`
- `top_k` по умолчанию: 3

### 4.2. Контрольные запросы и оценка retrieval
- Количество контрольных запросов: 12
- Метрики: `hit@3`, `recall@3`, `MRR@3`
- Результаты:  
  - `hit@3 = 1.000`  
  - `recall@3 = 1.000`  
  - `MRR@3 = 1.000` (нужный документ всегда на 1-м месте)
- Проблемных запросов не выявлено — база знаний очень чёткая.

## 5. Эксперимент и обновление базы знаний

- Эксперимент: сравнение `top_k=3` vs `top_k=5`.  
  Оба варианта дали `hit@k = 1.000`. Выбран `top_k=3` как более компактный для mini-RAG.
- Обновление: добавлено 5 новых документов (`bagging`, `boosting`, `batch_norm`, `dropout`, `rag`).
- Переиндексация: полная (пересчёт всех эмбеддингов + новый индекс).
- Результат: retrieval изменился во всех 5 тестовых запросах. Новые документы стали находиться корректно.

## 6. Mini-RAG

- Формирование контекста: топ-3 чанка с указанием источника.
- Генерация ответа: простой шаблон (берётся лучший чанк + перечисление остальных источников).
- Источники возвращаются вместе с ответом (`retrieved_sources`).
- Хорошо работает на прямых вопросах («gradient descent», «PCA», «RAG» после обновления).
- Проблемные случаи: запросы, для которых нет отдельного документа («vanishing gradient problem», детальное сравнение L1/L2).

## 7. Анализ

База знаний получилась удачной для демонстрации: высокая тематическая изолированность документов позволила получить идеальные метрики retrieval уже на старте. Это упрощает задачу, но хорошо показывает работу всего пайплайна.

Символьный чанкинг с перекрытием оказался достаточным. Основные ограничения mini-RAG связаны не с retrieval, а с:
- Неполнотой базы знаний (отсутствие отдельных документов на некоторые подтемы).
- Простотой генератора (шаблон не синтезирует информацию из нескольких источников).

Обновление базы знаний наглядно показало, как добавление новых документов сразу влияет на результаты retrieval — это одна из ключевых особенностей RAG-систем.

## 8. Итоговый вывод

Выполнен полный учебный RAG-пайплайн: чанкинг → эмбеддинги → FAISS → retrieval + оценка → обновление базы → mini-RAG.  
Главный вывод: качество конечного ответа сильно зависит от качества retrieval и полноты базы знаний. Даже простой шаблонный генератор даёт хорошие результаты, если retrieval работает идеально.
