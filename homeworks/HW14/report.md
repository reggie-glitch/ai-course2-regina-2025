# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- База знаний: **20 справочных статей по концепциям машинного обучения** (ML glossary) — каждый документ объясняет один термин/метод. Выбрана как тематически чёткая и удобная для retrieval-задачи.
- Чанкинг: символьный с перекрытием (`chunk_size=200`, `overlap=40`), итого **60 чанков** из 20 документов.
- Embedding-модель: `all-MiniLM-L6-v2` (sentence-transformers), векторы размерности 384, косинусное сходство через FAISS `IndexFlatIP` с нормализованными векторами.
- Базовая проверка retrieval на 12 контрольных запросах показала идеальный результат: `hit@3 = 1.000`, `recall@3 = 1.000`, `MRR@3 = 1.000` — ни одного промаха.
- После добавления 5 новых документов (bagging, boosting, batch norm, dropout, RAG) и переиндексации retrieval изменился во всех 5 тестовых запросах — новые источники стали находиться корректно.
- Mini-RAG успешно извлекает релевантные фрагменты и формирует ответ с указанием источников для всех протестированных вопросов.

## 2. Среда и воспроизводимость

## 3. База знаний и подготовка документов

- Предметная область: базовые концепции и алгоритмы машинного обучения (ML glossary)
- Источник документов: собственные текстовые описания, встроенные в код (без внешних зависимостей)
- Число исходных документов: 20 (+ 5 после обновления = 25)
- Число чанков после разбиения: 60 исходных → 75 после обновления
- Какой способ чанкинга использовался: символьный с перекрытием
- Основные параметры чанкинга: `chunk_size=200`, `overlap=40`
- Комментарий: База знаний состоит из коротких тематически изолированных статей — каждый документ описывает ровно одну ML-концепцию. Это удобная структура для retrieval: запрос про конкретный метод однозначно указывает на один релевантный документ. Тексты достаточно длинные (400–500 символов), чтобы разбиваться на 2–3 чанка с сохранением смысла. Символьный чанкинг с перекрытием прост и воспроизводим; перекрытие в 40 символов снижает риск разрыва важного контекста на границах чанков. База компактна (25 документов после обновления), что позволяет выполнить весь pipeline на CPU без задержек.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Какая embedding-модель использовалась: `all-MiniLM-L6-v2` (sentence-transformers) — компактная (384 измерения), быстрая, хорошо работает на семантическом поиске
- Какая метрика сходства использовалась: косинусное сходство (через `IndexFlatIP` с нормализованными векторами)
- Какой индекс FAISS использовался: `IndexFlatIP` — точный перебор, достаточен для 60–75 векторов
- Какой `top_k` использовался по умолчанию: 3
- Что показали первые примеры retrieval: модель уверенно находит нужные документы; для релевантных чанков оценки сходства в диапазоне 0.60–0.78, для нерелевантных — заметно ниже (0.47–0.51). Например, запрос «What is overfitting and how to prevent it?» вернул все три слота из `doc_11_overfitting` с оценками 0.765, 0.726, 0.561.

### 4.2. Контрольные запросы и оценка retrieval

- Сколько контрольных запросов использовалось: 12
- Что считалось релевантным ответом: наличие ожидаемого документа (`expected_source`) среди top-3 извлечённых источников
- Какие метрики retrieval считались: `hit@3`, `recall@3`, `MRR@3`
- Итоговый `hit@3`: **1.000** (12 из 12)
- Итоговый `recall@3`: **1.000**
- Дополнительные метрики: `MRR@3 = 1.000` — во всех случаях нужный документ оказался на первом месте (`rank_of_first_relevant = 1`)
- Какие запросы оказались самыми простыми: все 12 запросов — промахов не было; наиболее уверенные результаты у запросов с уникальными ключевыми словами: «gradient descent», «PCA reduce dimensionality», «kernel trick in SVM»
- Какие запросы оказались самыми проблемными: проблемных запросов не выявлено; потенциально сложными могут быть запросы с пересекающимися концепциями (например, «sentence embeddings» vs «transformer embeddings»), но в данном наборе они были разрешены корректно

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- Какой параметр сравнивался: `top_k`
- Какие два варианта сравнивались: `top_k=3` vs `top_k=5`
- Как изменилось качество retrieval: оба варианта дали `hit@k = 1.000` и `recall@k = 1.000` — база знаний достаточно чёткая, чтобы нужный документ попадал в top-3
- Какой вариант выбран как основной: `top_k=3` — при одинаковом качестве retrieval он даёт более компактный и сфокусированный контекст для mini-RAG
- Что именно было обновлено в базе знаний: добавлены 5 новых документов: `doc_21_bagging`, `doc_22_boosting`, `doc_23_batch_norm`, `doc_24_dropout`, `doc_25_rag`
- Как была выполнена переиндексация: полная переиндексация — пересчёт эмбеддингов для всех 75 чанков и создание нового `IndexFlatIP`
- Что изменилось в retrieval после обновления: retrieval изменился во всех 5 тестовых запросах (`changed=1` для всех). Например, запрос «What is bagging in ensemble learning?» раньше возвращал `doc_04_random_forest | doc_02_logistic_regression`, а после — `doc_21_bagging | doc_21_bagging | doc_04_random_forest`. Запрос «What is Retrieval-Augmented Generation (RAG)?» сменился с `doc_19_transfer_learning | doc_09_rnn` на `doc_25_rag | doc_25_rag | doc_19_transfer_learning`.

## 7. Анализ

Выбранная база знаний оказалась очень удачной для учебного эксперимента: тематически чёткая структура (один документ — одна концепция) позволила получить идеальные метрики retrieval (`hit@3 = 1.000`) уже на первом запуске. Это одновременно достоинство и ограничение: в реальных задачах документы перекрываются по содержанию, что делает retrieval значительно сложнее.

Символьный чанкинг с перекрытием показал себя достаточным для данной базы. При `chunk_size=200` и `overlap=40` каждый документ разбился на 2–3 чанка, что сохранило смысловой контекст. Более мелкий чанкинг мог бы разбить ключевые определения на части и снизить точность поиска.

Формальная оценка retrieval оказалась полезной даже при идеальных результатах: она дала чёткое численное подтверждение работоспособности системы. `MRR@3 = 1.000` означает, что нужный документ всегда оказывается на первом месте — у системы нет проблем с ранжированием. Эксперимент с `top_k` показал, что для данной базы оба значения эквивалентны, и выбор `top_k=3` обоснован соображениями компактности контекста для генератора.

Обновление базы знаний наглядно продемонстрировало ключевой принцип RAG: retrieval изменился во всех 5 тестовых случаях. Запрос про RAG до обновления возвращал `doc_19_transfer_learning` — тематически близкий, но неточный; после — `doc_25_rag` с релевантным определением. Запрос про bagging ранее указывал на `doc_04_random_forest`, после добавления `doc_21_bagging` — находит точный ответ.

Наиболее показательные ошибки mini-RAG связаны не с retrieval, а с неполнотой базы знаний: «vanishing gradient problem» не имеет отдельного документа, поэтому ответ получается частичным. Вторая категория ошибок — ограничения шаблонного генератора: он не умеет синтезировать информацию из нескольких чанков и просто возвращает первый найденный фрагмент. Дополнительное ограничение — дублирование чанков в top-k: иногда все три слота занимали чанки одного документа (например, `doc_05_svm | doc_05_svm | doc_05_svm`), что избыточно и не даёт дополнительной информации.

## 8. Итоговый вывод

Как базовый retrieval-конвейер для семантического поиска по небольшой базе знаний оптимален следующий стек: `all-MiniLM-L6-v2` для эмбеддингов, `FAISS IndexFlatIP` с нормализованными векторами, `top_k=3`, символьный чанкинг с перекрытием (`chunk_size=200`, `overlap=40`). Главное, что я понял про качество retrieval: оно определяется тремя факторами — качеством embedding-модели, параметрами чанкинга и полнотой базы знаний; при этом полнота базы критична — даже идеальный retrieval не поможет, если нужного документа в базе нет. Главная связь retrieval и mini-RAG: генератор полностью ограничен качеством найденного контекста — хороший retrieval позволяет даже простому шаблонному генератору давать полезные ответы, а плохой retrieval делает любой генератор бесполезным независимо от его сложности.
