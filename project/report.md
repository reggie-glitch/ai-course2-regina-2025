# Отчёт по проекту: IDS — Классификатор сетевых вторжений

## 1. Постановка задачи

**Предметная область:** информационная безопасность, системы обнаружения вторжений (IDS).

**Пользователь:** аналитик SOC, который обрабатывает тысячи сетевых событий в день.

**Задача:** по 41 признаку сетевого соединения определить — нормальный трафик или атака.

**Тип задачи:** бинарная классификация (normal = 0, attack = 1).

**Целевая метрика:** F1-score (weighted). Выбор обоснован дисбалансом классов и двойной ценой ошибок:
- пропустить атаку (FN) — критично для безопасности;
- ложное срабатывание (FP) — перегружает аналитика.

---

## 2. Данные

**Датасет:** NSL-KDD — улучшенная версия KDD Cup 1999.  
Источник: https://www.unb.ca/cic/datasets/nsl.html

**Ключевые особенности:**
- Нет дублирующих записей (в отличие от оригинального KDD'99)
- 125 973 обучающих и 22 544 тестовых записей
- 41 признак: числовые + 3 категориальных (protocol_type, service, flag)
- Метки: 23 типа атак, сгруппированных в бинарную метку

**Предобработка:**
1. Метки: `normal` → 0, все остальные → 1
2. Категориальные признаки → LabelEncoder (fit только на train+test для полного охвата)
3. Числовые признаки → StandardScaler (fit только на train)

**Дисбаланс классов (train):**
- normal: ~53 900 записей (~43%)
- attack: ~67 300 записей (~57%)

Дисбаланс умеренный, специальных техник (oversampling, class weights) не потребовалось.

---

## 3. Эксперименты

### 3.1 Разбивка данных

Использовали готовый train/test split датасета NSL-KDD (стандартный для сравнения с литературой).

### 3.2 Baseline — Logistic Regression

Первая модель для оценки нижней границы качества. Работает на масштабированных признаках.

**Результат:** F1 ≈ 0.98, ROC-AUC ≈ 0.99

Вывод: задача достаточно линейно разделима — линейная модель уже даёт высокий результат.

### 3.3 Decision Tree

Без масштабирования (деревья инвариантны к масштабу), `max_depth=15`.

**Результат:** F1 ≈ 0.990, ROC-AUC ≈ 0.990

Вывод: чуть лучше baseline, но уступает ансамблям.

### 3.4 Random Forest

100 деревьев, `max_depth=20`, параллельное обучение (`n_jobs=-1`).

**Результат:** F1 ≈ 0.998, ROC-AUC ≈ 0.9995

Вывод: значительный прирост за счёт усреднения деревьев.

### 3.5 XGBoost (финальная модель)

Градиентный бустинг, 200 итераций, `learning_rate=0.1`, `max_depth=6`.

**Результат:** F1 ≈ 0.998, ROC-AUC ≈ 0.9997

Вывод: лучший результат по обеим метрикам, выбран как финальная модель.

## 5. Анализ ошибок

Типичные ошибки финальной модели (по confusion matrix):
- Ложноположительные (normal → attack): редкие соединения с нестандартными параметрами
- Ложноотрицательные (attack → normal): атаки типа R2L/U2R, которые мало представлены в train

Для снижения FN по редким классам атак — следующий шаг: перейти к мультиклассовой классификации с отдельным балансированием по типам атак.

---

## 6. Важные признаки (Feature Importance, XGBoost)

Топ-5 по важности:
1. `src_bytes` — число байт от источника
2. `dst_bytes` — число байт от получателя
3. `serror_rate` — доля SYN-ошибок
4. `count` — число соединений к тому же хосту за 2 сек
5. `logged_in` — был ли выполнен успешный вход

Это соответствует интуиции: атаки DoS характеризуются аномальным числом пакетов и ошибок.

## 7. Архитектура сервиса

```
Client → POST /predict → FastAPI (app.py)
                              ↓
                        predict.py
                         └─ preprocess_input()   ← encoders + scaler
                         └─ model.predict_proba() ← XGBoost
                              ↓
                        JSON response
```

**Наблюдаемость:**
- Логирование каждого запроса (метод, путь, статус, время)
- In-memory счётчики: `requests_total`, `predictions_normal`, `predictions_attack`, `errors_total`
- `/health` эндпоинт для мониторинга

---

## 8. Ограничения и дальнейшие шаги

**Текущие ограничения:**
- Модель обучена на NSL-KDD 1999 года — современные атаки (например, web-shell, cryptominer) не представлены
- Статические пороги классификации — не адаптируются к дрейфу данных
- In-memory метрики — не персистентны (сбрасываются при перезапуске)

**Следующие шаги:**
1. Периодическое переобучение на новых данных (MLOps)
2. Подключение Prometheus + Grafana для персистентных метрик
3. Мультиклассовая версия для детального анализа типов атак
4. A/B тестирование моделей на продакшн-трафике
